# HILO 最优版总结 · strat90 user 模式 · 2026-06-25

> 最高分版本:`strat90_hilo` = **70/90 = 78%**（telecom user 模式，gpt-5.4 用户模拟器）。
> 配置栈:`REPLAN + GROUND + ELICIT + SEQ + CLOSE(常开) + PROV + UNLATCH + REPLY + HILO`
> （不含 WATCHDOG——实测纯负；不含 DEFER——已回退；SHOTGUN/SAGE/SUPERVISOR/DIAG 关）
>
> 家族增量:mobile_data 21→**29/30** · mms 9→11 · service 30/30（无回归）。

---

## 一、三层架构 + 两个 LLM 的分工

确定性 **Python 控制层 + 小模型主力 + 大模型稀疏纠错**，外加一个**大模型扮演的用户**（属于环境，不是 agent）。

| 层 | 是谁 | 干什么 | 是否 LLM |
|---|---|---|---|
| **控制层** | 纯 Python（schema 走查） | 选哪个子任务、何时收尾/放弃、REPLAN 重排、PROV 门控、各种 guard | ❌ 只路由，不理解内容 |
| **主力模型** | 本地 **qwen3-8b**（@127.0.0.1:9000） | 干**每一步**的内容活（3 个角色，见下） | ✅ 小模型 |
| **稀疏纠错** | **gpt-5.4**（经 .env 代理） | 只在**载荷抽取**处复核/重做 8B 的解析 | ✅ 大模型，占 **9.4%** token |
| **用户** | **gpt-5.4**（user_simulator） | 模拟客户、操作设备、回报结果 | ✅ 环境，非 agent |

关键:**8B 是 workhorse（90.6% 的 token），大模型只补最容易错的那 9.4%**。这是整套方案的效率主张。

> 结构性背景:telecom user 模式下，所有 device 动作（run_speed_test / grant_app_permission / toggle_* …，共 59 个）都是**用户工具**，agent 调不到；agent 只有账户/计费工具。所以排障 = agent 多轮**指导用户操作手机 + 解读模糊回复**。本套 schema 最初按 solo 文档写（"agent 直接操作设备"），user 模式的修复都在补这个接缝。

## 二、LLM 各自的角色与机制（核心）

### 小模型 8B —— 每步都在跑，3 个角色

| 角色 | 触发 | 机制 |
|---|---|---|
| **MacroInitializer** | 每 episode 1 次 | ticket → 初始 `task_state` |
| **Executor** | 每步 1 次 | 在当前子任务允许的工具内，生成**一条给用户的自然语指令**（device 动作是 user 工具，8B 不能直接调，只能指导用户） |
| **StateUpdater** | 每条用户回话 1 次 | 把客户的自由文本回话解析成 `patch_ops`（状态字段更新） |

### 大模型 gpt-5.4（HILO）—— 稀疏纠错，只接管最危险的"解析"

**为什么是它**:离线诊断（gpt-5.4 当 oracle，80 条高后果抽取）证明 **StateUpdater 解析是瓶颈** —— 高后果区 **~19% 的抽取有后果地错**（把"can't send MMS"写成 service_status、从失败回话读出 resolved、漏抽同句 co-field、值分级错），而且是**自信地错**:8B 自洽采样只能逮 **13%**，所以 SAGE 式"模型不确定性=采样方差"在这里失效。

**机制（两层路由）**:8B 先解析每一条；然后——

- **Tier-1（总是上大模型）**:终末 `VERIFY_SPEED/VERIFY_MMS/VERIFY_SERVICE/RETEST_MMS` + mms 修复区 `FIX_STORAGE_PERM/FIX_SMS_PERM/FIX_NETWORK_MODE_MMS/FIX_WIFI_CALLING` —— 错一次就 phantom-close 或漏解决的载荷抽取，大模型**直接重抽并覆盖** 8B 结果。
- **Tier-2（别处只在可疑时）**:廉价结构门（写了 `resolved`、从纯提问里幻觉、富信息回话却空写、失败回话却写成功值）标记 → 才上大模型。
- 每集预算上限 `SCHEMAFLEX_HILO_MAX=12`；token sidecar 拆 `updater`(8B) vs `updater_hi`(大模型)。
- 实测:**350 次调用 ≈ 3.9 次/任务，占全部 agent token 的 9.4%**。

> 离线门测量结论:结构门 ~47% 命中 / 3% 误报；自洽采样 13% 命中（弱）；并集 ~60% 召回。剩 40% 漏报是语义级（co-field 漏抽、值分级），**不重抽就检测不出**——所以靠 Tier-1"按后果路由"覆盖，而非靠门。

## 三、各 lever（技术）做什么 + 是 LLM 还是 Python

| Lever | 治什么 | 机制 | 层 | 是否真出力 |
|---|---|---|---|---|
| **REPLAN** | 静态走查把诊断/修复顺序写死 | 按因果链(produces/consumes)动态重排，修复先行、剪死分支 | Python | ✅ |
| **GROUND** | agent↔user 指令接缝糙 | 一次一动作、字面拼 tool/args、禁美化命名、检查类只 LOOK&report | Python（约束 8B 输出） | ✅ |
| **ELICIT** | 缺只有用户知道的上下文 | give-up 前问一句 yes/no 补 null 字段（pay_allowed / is_abroad） | Python+8B | ✅（overdue_bill） |
| **SEQ** | 只读门→写 的序列死锁 | 跳过已跑只读门、推进到终末写；末位写未跑不判 done | Python | ✅ |
| **CLOSE** | 目标已达但无人收尾→撞 max_steps | success_when_all 满足即主动收尾（护栏:必须真 dispatch 过 mutating 工具） | Python | ✅ 常开 |
| **PROV** | goal 字段被 FIX 步乐观早写→phantom resolved latch | **action 级 provenance**:speed_test/can_send_mms 只能由其观测工具(run_speed_test/can_send_mms)那次回话写 | Python | ✅✅ **mobile_data +8 的主因** |
| **REPLY** | 字面 app 名僵局 | 把用户上一句注入指令措辞 + 普通走查也钉死字面 "messaging" | Python+8B | ✅ mms |
| **UNLATCH** | resolved 误锁死冻结 | 早闭后用户仍说话→清 resolved 重开验证 | Python | ⚪ 本轮 **0 次触发**（PROV 开后无 phantom 可解，无害冗余） |
| **HILO** | 8B StateUpdater 解析错 | 稀疏大模型重抽载荷抽取 | **8B+大模型** | ✅✅ **精度主因** |

**真正搬动分数的是 PROV（治 mobile_data verify-stall）和 HILO（治 8B 解析错）**；REPLY 帮了 mms；其余 lever 是地基。

## 四、为什么这么分（效率机制）

- 任务成功 ≈ 每步内容正确率 `p` 的 `p^k`。深任务 k≈15–20，8B 的 `p` 不够 → 深 mms 坍塌。
- **Python 层只能减 k、剪死路、堵系统性错**（PROV 堵 phantom 就是），**改不了 p**（它不读内容）。
- **要提 p 必须靠 LLM**，但全用大模型就丢了"小模型/效率"卖点 → **HILO 只把大模型插在最容易错、错了最致命的载荷抽取上**（9.4% token）。
- 效率主张的诚实口径:**相对全大模型 agent，大模型 token 少 ~10.6×**；**不是"近免费"**（8B 自托管≈免费，$ 几乎都在那 9.4%）。可主张:"8B + 9.4%-token 的稀疏大模型监督，逼近大模型精度、只花 ~1/10 的大模型成本"。

## 五、诚实边界（这一版到不了 100%）

- 70/90 的剩余失败**几乎全是深 mms（4–8 fault）的步数预算问题，不是精度**:它们在轨迹 83–96% 处还在修新故障，被 100 步砍断（11/12 个 max_steps 实证）。
- **HILO 把精度兜住了，深 mms 要靠调大 MAX_STEPS 兑现**——但治标硬顶 **~87–92%**（7/20 是 user_stop，用户对过长会话 OOS，加步数救不了；且更长会话反而更易 OOS）。
- 试过的步数治本 **DEFER 失败并已回退**:它跳过修复内重测，导致终末验证被一次早期诊断的 stale 值提前满足 → 最后那次验证整段没跑 → agent 修完后无事可做就转人工 → 判 0（把 mobile_data 25→13 砸了）。教训:**修复内重测是承重的**（每修必验 + 防止终末验证被 stale 诊断值满足），不能删；正确的步数治本得"终末验证在修复落地后强制重开重跑"或"批量诊断 CHECK"，且必须在能实跑的环境里迭代验证（纯离线单元测试不够，会漏掉真实交互的坑）。

## 六、复现命令

```bash
cd /root/Project/SchemaFlex/tau2-bench
# 最优版（70/90 那套）
SCHEMAFLEX_REPLAN=1 SCHEMAFLEX_GROUND=1 SCHEMAFLEX_ELICIT=1 SCHEMAFLEX_SEQ=1 \
SCHEMAFLEX_PROV=1 SCHEMAFLEX_UNLATCH=1 SCHEMAFLEX_REPLY=1 SCHEMAFLEX_HILO=1 \
NUM_TRIALS=1 \
scripts/runUser.sh strat90_hilo --task-set-name telecom_strat90 \
  --max-concurrency 1 --max-errors 200

# 调大步数做最后测试（深 mms 在 200 步内兑现）
#   在上面基础上加 MAX_STEPS=200，run 名换 strat90_hilo_s200
```

> 注意:HILO 让 agent 侧也打 gpt-5.4，叠加 user-sim 的并发调用会把代理打爆 → **务必 `--max-concurrency 1`（或 2）、`--max-errors 200`**，否则 tau2 会因连环报错满 `--max-errors`(默认10) 而早退成空结果。

## 七、一句话

确定性 schema 走查（定步骤）+ 8B 干所有内容活（廉价主力）+ gpt-5.4 只补 9.4% 最致命的解析（稀疏纠错）+ 一组针对实证失败点的编排 lever（PROV/REPLAN/GROUND…，减步/堵错）。**贡献是"廉价小模型 + 稀疏高模监督逼近高精度、只花零头的大模型成本"，不是拼绝对精度。**
