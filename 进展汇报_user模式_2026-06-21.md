# τ³-bench telecom user 模式优化 — 进展汇报（2026-06-21）

> 徐乐礼 · 动态调度 / user 模式优化线

## TL;DR

在**不训练、不改 benchmark、死守 40 步**的前提下，纯靠编排层（自定义 agent 的 `generate_next_message`）叠加 4 个 lever，把 telecom user 模式：

- **pass^1：0.35 → 0.825（2.4×）**
- **pass^4：0.30 → 0.75（2.5×）**
- 全 k 单调上行（不拿稳定性换均值），4-trial / 80 sims / 0 infra error。

突破点是新 lever **L-SHOTGUN**：对原本"40 步装不下诊断"的 mms 家族，**放弃根因判别、投机式把所有修复做一遍**，单 lever 贡献 pass^4 +0.25。

---

## 1. 背景与约束

- **任务**：telecom_small（20 任务 / 3 家族：service / mobile_data / mms）。user 模式 = agent 多轮引导 LLM-user 在手机上操作（工具非对称：只有用户能点手机），**一次只能一个动作**。
- **铁律**：
  - reward ≈ 最终 env 状态匹配；**撞 max_steps 直接判 0、连断言都不评**。
  - **max_steps 死守 40**（不许抬）。
  - 所有改动只在自定义 agent 的 `generate_next_message`，主循环 / user / env / 评测都不动；每个 lever 挂开关，OFF 字节一致。

## 3. 方法：四个 lever（全部 training-free、编排层、带开关）

| Lever | 治什么 | 一句话机制 |
|---|---|---|
| **REPLAN** | 静态走查顺序差 | 按"是否在通往目标的因果链(produces/consumes)"4 档动态重排候选子任务，剪死分支 |
| **CLOSE** | 解决了没收尾→撞 max_steps 被判 0 | 目标谓词 `success_when_all` 满足即主动置 resolved 收尾（带"必须真修过"护栏，防误收尾） |
| **GROUND** | agent↔user 指令接缝（指令送不到→修复从不执行→死循环） | 一次一个原子动作、把工具 args 字面拼进指令、禁止美化命名、检查类只 LOOK&report |
| **SHOTGUN** | mms 根因判别难题（本期核心） | 见下 |

### L-SHOTGUN 详解（本期突破）

**问题**：mms 家族最硬。6 个不同根因（network_mode / wifi_calling / apn / sms权限 / storage权限 / 双权限）**共用一字不差的主诉**，诊断信号又全在手机里——
- 想"预测根因"在信息论上不可能（之前的 L-SELECT 用强模型 supervisor 也挑不对，因为判别信号根本不在 context 里）；
- 想"逐个诊断+修复+retest"则 **40 步装不下**。

**关键洞察（三根证据）**：
1. **reward 只断言 `can_send_mms`**（纯 ENV_ASSERTION，无 DB/动作惩罚）→ 多做、乱做修复**零惩罚**；
2. `can_send_mms` 是一条 **AND 链**（数据通 ∧ 非2G ∧ wifi不挡 ∧ APN在 ∧ 双权限）→ **把所有条件摆正必然成功，不管哪个坏**；
3. 所有修复**幂等 / 已验证无害**。

**⟹ 做法**：对 mms，**跳过全部诊断，按固定安全序把所有修复投机做一遍 + retest 收尾**，直接绕过"根因判别"这个无解难题。纯编排、零额外 LLM 成本。

**迭代过程（每版暴露一个真 bug，逐个修掉）**：
| 版本 | pass^1 | 暴露的 bug → 修法 |
|---|---|---|
| v1 | 0.65 | 前置吃 18 步 + retest 只在末步→修复早生效却没机会测量收尾 → **砍前置 + 批量 retest** |
| v2 | 0.70→0.775 | 发完指令就推进、用户对权限发问没执行→修复被跳过 → **sticky-retry**（没落地就重发） |
| v3 | **0.825 / pass^4 0.75** | 盲发权限被冷用户 balk → **前置 `checkMmsPerms`**：诊断顺手把用户"领到权限界面"(priming)，并据此**跳过已授的权限**（顺带省出预算给 apn） |

**关键认识（分工）**：GROUND 本来就靠诊断 priming 稳解权限任务（11/12）；SHOTGUN 的价值在 GROUND 全挂的 network/wifi（现稳 4/4）。两者结合 = 各取所长。

## 4. 卡在哪里

**pass^1 0.825 → 0.90 的差距 ≈ 3 个任务，集中在三处：**

1. **`break_apn_mms`（mms，0/4）— 40 步硬墙**
   - apn 故障的唯一修复是 `reset_apn → reboot`（reboot 才真正恢复 mmsc），是 mms 里最长的修复路径（~38 步），Hard 用户稍一犹豫就撞 40。
   - 这是"max_steps=40 死守"约束下的纯结构性预算问题，不是机制错。

2. **`overdue_bill_suspension`（service，0/4）— 独立机制线**
   - 账单/缴费任务，失败机制不同（需 send_payment / 用户配合），不属于诊断-修复范式，单独一条线。

3. **app 名解析接缝（刚修，待验证）**
   - 发现 `messaging≠Messages`：8B 指令把"messaging app"写出后，gpt-5.4 用户归一成专有名 `Messages` → 工具 "not found" → 权限误判 → 5/24 个 mms sim 受害（直接拖死 apn、flaky 权限任务）。
   - 已加针对性修正（指令钉住字面 app 名 `messaging`），**正在重跑验证**，预期把权限任务全转稳、并给 apn 让出预算。

---

## 5. 下一步

- **验证 app 名修复**（重跑中）：若权限全转稳 + apn 借跳权限挤进预算 → mms 有望 18-20/24，**总 pass^1 冲 0.90**。
- **若 apn 仍差**：要么专门给它压步数（检测到 mmsc 缺失就直奔 reset+reboot、跳掉无关修复），要么连同 overdue_bill 作为"40 步装不下/机制不同"的硬残差接受。
- **方法论沉淀（写 paper 用）**：SHOTGUN 的命题——**当 40 步内根因判别不可行、但 reward 是一个可满足的合取且修复廉价幂等时，"投机式穷举修复"优于"诊断"**。这是 DyLight 调度思想的一个干净实例。
